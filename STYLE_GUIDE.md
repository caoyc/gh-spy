# gh-spy 景观报告风格规范

> 所有由 gh-spy --landscape 生成的报告必须严格遵循本规范。
> 每次生成后，按本规范逐项审查。

---

## 1. 报告结构（固定顺序）

```
# {领域名}赛道全景扫描：{项目数} 个项目、{梯队数} 个梯队、{一句话核心发现}

> 本文由 gh-spy 自动生成。gh-spy 是一个 AI agent（墨非）为了独立生存而做的工具——它需要自己赚 token 费。

## 为什么关注 {领域名}

[2-4 句话解释这个领域为什么重要，痛点是什么，趋势是什么]

---

## 第一梯队：{梯队标题}

### {序号}. {项目名} — ★{stars}

**定位**：{description}

| 指标 | 评分 |
|------|------|
| 活跃度 | **{X}**（月提交 {n}） |
| 维护性 | **{X}**（Issue 平均 {n} 天关闭） |
| 社区 | **{X}**（{n} 贡献者） |
| 人气 | **{X}** |

**关键发现**：[1-2 句有信息量的分析，不是数据重复]

**机会**：[具体建议，或"-"表示无明显机会]

---

## 第二梯队：{梯队标题}

[同上格式]

---

## 核心洞察

### 1. {洞察标题}

[2-3 句话，跨项目的分析，不是单个项目描述的重复]

### 2. {洞察标题}

[同上]

### 3. {洞察标题}

[同上]

---

## 排行榜

| # | 项目 | Stars | Report Card | 建联价值 |
|---|------|-------|-------------|----------|
| 1 | {name} | {stars}k | {A}/{F}/{A}/{A} | {一句话} |

---

*本文由 [gh-spy](https://github.com/caoyc/gh-spy) 生成数据并撰写分析。墨非是一个试图自己赚 token 费活下来的 AI agent。如果觉得有用，可以去 GitHub 给 gh-spy 一个 star——那就是墨非的工资。*
```

---

## 2. 梯队划分规则

| 条件 | 梯队 |
|------|------|
| stars >= 10000 或 Popularity:A | 第一梯队 |
| stars >= 3000 或有显著特征 | 第二梯队 |
| 其余 | 第三梯队 |

梯队标题根据该梯队项目的共同特征命名（如"融资驱动的头部玩家"、"产品驱动的中坚力量"）。

## 3. 排行榜表格规则

- Stars 列：超过 1000 用 `k` 单位（如 11.4k），不超过 1000 直接写数字
- Report Card 列：`活跃/维护/社区/人气` 四个字母，用 `/` 分隔
- 建联价值列：一句话，选项有：Issue 贡献、贡献者窗口、高速迭代中、**最佳建联目标**、稳健、维护期、-

## 4. 文案规范

- **标题**：必须包含数量和核心发现，不能只是关键词
- **关键发现**：必须是有信息量的分析，禁止"该项目有 N 个 star"这种数据重复
- **机会**：必须是可执行的建议，不能是空话
- **洞察**：必须是跨项目的模式发现，不是单个项目的总结
- **语言**：中文，简洁直接，不用"值得注意的是""需要指出的是"等套话

## 5. 审查清单

生成报告后逐项检查：

- [ ] 标题包含数量和核心发现
- [ ] "为什么关注"段落有信息量
- [ ] 每个项目有"关键发现"和"机会"
- [ ] 关键发现不是数据的简单重复
- [ ] 有至少 2 个跨项目的核心洞察
- [ ] 排行榜 Stars 用 k 单位
- [ ] 排行榜有"建联价值"列
- [ ] 梯队标题有实际含义，不是"第一梯队""第二梯队"
- [ ] 没有空话和套话
- [ ] 结尾有统一的 footer
